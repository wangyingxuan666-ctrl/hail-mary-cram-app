## 一、项目概述

**项目名称**：Hail Mary Cram（临时抱佛脚）—— AI 驱动的考前突击学习平台

**项目定位**：将一套成熟的 AI 备考方法论（"课件做索引、真题做主线、讲解奶奶都能听懂"）从 Claude Code 插件改造为独立的 SaaS 级 Web 应用。

**核心价值**：帮助学生在考前 3 天内，通过 AI 自动分析历年真题考点频率、按优先级逐题讲解、生成考场可默写的速记卡，实现"临时抱佛脚"式的高效突击。

**个人角色**：独立全栈开发（Full-Stack Developer），负责从需求分析、架构设计到前后端开发、部署验证的全部环节。

**开发周期**：1 天完成从零到可运行的全栈原型（含 20+ API 端点、5 个前端页面、AI 流水线集成）。

## 三、架构设计能力体现

### 3.1 从"插件"到"独立应用"的产品化思维

原始项目是一个 Claude Code 的 SKILL.md 配置文件（纯 Markdown 提示词），我完成了以下产品化改造：

- **模块拆分**：将单文件 SKILL.md 中的方法论解耦为独立的 Prompts 层（系统提示词 / 频率分析提示词 / 6 步讲解提示词）
- **服务抽象**：识别出文档加载、向量存储、频率分析、讲解生成、RAG 检索、记忆追踪、文档导出共 7 个独立服务
- **数据建模**：设计 4 组 Pydantic 模型（Document / Exam / Chat / Memory），覆盖文件 → 分析 → 对话 → 追踪的完整数据流
- **API 设计**：规划 5 组 RESTful 路由（Upload / Analysis / Chat / Export / Memory），共 20+ 个端点

### 3.2 核心 AI 流水线设计

```
用户上传课件 + 真题
      ↓
文档加载 → 文本提取 → RecursiveCharacterTextSplitter(1200/200) → FAISS 向量化存储
      ↓
LLM 扫描所有真题 → 提取每道题的考点 + 年份 → 聚合频率 → 按 ≥50% / 30-49% / 20-29% / <20% 分级着色
      ↓
用户选择题目 → RAG 检索 Top-K 相关课件片段 → 组装 3 段式 + 6 步法 Prompt → DeepSeek 流式生成
      ↓
SSE 逐字推送到前端 → react-markdown 实时渲染 → 对话历史持久化到本地 JSON
```

### 3.3 Prompt Engineering（提示词工程）

这是项目的核心竞争力——将一套经过实战验证的教学方法论编码为 LLM 可稳定执行的系统提示词：

- **3 段式外层结构**：① 中文白话讲解（零基础友好） → ② 中英双语考场答案（含 6 步内层） → ③ 中文失分提醒（反例 + 易混淆点）
- **6 步内层结构**：一句话说清 → 背景故事 → 生活类比 → 举一反三 → 抓重点 → 失分提醒
- **硬约束规则**：禁止邀请试答、禁止一次讲多题、禁止照搬定义、禁止范文长段落、图片缺失时主动要求截图
- **Few-shot 示例**：在提示词中嵌入正例和反例，确保 LLM 输出风格稳定

## 四、技术难点与解决方案

### 4.1 流式 AI 响应的实时渲染

**难点**：DeepSeek API 返回的 SSE 流需要逐 token 解析，同时前端要支持 Markdown 的增量渲染（代码块、表格等可能跨越多个 chunk）。

**方案**：
- 后端使用 `openai.ChatCompletion.create(stream=True)` 逐 chunk yield
- 通过 FastAPI `EventSourceResponse` 封装为标准 SSE 格式
- 前端使用 `fetch` + `ReadableStream` + `TextDecoder` 手动解析 SSE 帧
- `react-markdown` 对不完整 Markdown 也能部分渲染，配合 `useState` 追加式更新实现打字机效果

### 4.2 扫描件 PDF 的文字提取

**难点**：许多课件和试卷是扫描图片，`PyPDFLoader` 提取不到文字层。

**方案**：
- 实现三级回退策略：`PyPDFLoader` → `pdfplumber`（支持 OCR 文本层提取） → 失败则提示用户截图
- 在系统提示词中编码"图片缺失时主动要求截图"的规则，避免 AI 凭空推测几何图形

### 4.3 LLM 输出的结构化解析

**难点**：频率分析需要 LLM 输出 `[{topic_name, year, question_number}]` 的 JSON 数组，但 LLM 可能包裹 Markdown 代码块或添加额外解释。

**方案**：
- 实现三层 JSON 解析器：直接 `json.loads()` → 正则提取 Markdown 代码块 → 正则提取裸 JSON 数组
- 每种解析失败时自动降级到下一层，确保鲁棒性

### 4.4 前后端类型一致性

**难点**：Python Pydantic 模型和 TypeScript 接口需要保持同步。

**方案**：
- 后端 4 组 Pydantic 模型（document.py / exam.py / chat.py / memory.py）作为单一数据源
- 前端 `types/index.ts` 手动镜像，但通过统一的 REST 契约（JSON 序列化）保证运行时兼容

## 六、项目亮点（面试话术参考）

1. **独立完成从方法论到产品的全链路**：从阅读开源插件的 SKILL.md 提示词文件开始，理解核心教学逻辑，设计数据模型、API 架构、AI 流水线，最终交付可运行的全栈 Web 应用。

2. **AI 工程化能力**：不只调用 API，而是围绕 LLM 构建了完整的工程体系——文档预处理 → 向量化 → 语义检索 → Prompt 组装 → 流式输出 → 结果持久化。

3. **Prompt Engineering 实战**：将教学专家的方法论编码为包含正反例、硬约束、结构化输出的系统提示词，是典型的"领域知识 + LLM 适配"案例。

4. **技术选型有据**：每个技术选择都有明确的 trade-off 分析（如 FAISS vs ChromaDB、SSE vs WebSocket、JSON 文件存储 vs SQLite），体现工程判断力。

5. **快速原型能力**：1 天内完成 20+ API 端点、5 个前端页面、完整 AI 流水线——证明从零到可运行交付的执行效率。
